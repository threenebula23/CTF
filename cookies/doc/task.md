Задание 2 — Cookies

150

Мой знакомый разработал платформу онлайн-магазина. Он попросил провести blackbox-тестирование. Проверь безопасность cookie, возможно, ты сможешь получить права администратора.

Ссылка на сайт: http://193.41.142.9:8001/shop/login

Краткое описание решения

При логине под пользователем в браузере обнаруживается cookie (ключ secret), значение которого — base64-строка. После декодирования base64 получается либо непосредственно ASCII-строка с JSON-объектом, либо строка, содержащая двоичные представления байтов (группы из 8 бит, разделённые пробелами). Этот JSON содержит поля username и is_admin, например:

{"username":"user", "is_admin":"False"}

Если сервер не подписывает и не проверяет целостность cookie (нет HMAC/подписи), то изменив поле is_admin на "True" и записав обратно модифицированное значение cookie, можно получить права администратора и доступ к защищённым страницам.

Теория для решения

1) Cookie — это пара ключ=значение, которую сервер отправляет клиенту через заголовок Set-Cookie. Браузер сохраняет её и при последующих запросах автоматически отправляет в заголовке Cookie.
2) В cookie может храниться сам объект с данными пользователя. Если эти данные важны (права, баланс и т.д.), сервер обязан гарантировать их целостность и конфиденциальность: подписывать (HMAC) или шифровать, либо хранить только идентификатор сессии, а сам объект — на сервере.
3) В нашем случае сервер хранит (в явном виде) JSON в виде base64-строки без подписи. Это уязвимость: клиент может модифицировать значение и получить повышенные права.

Что именно нужно сделать

- Получить значение cookie secret после логина под user.
- base64-decode: base64_decode(cookie_value) → raw bytes.
- Если raw bytes корректно декодируются в UTF-8 и дают JSON — парсим JSON.
- Если raw bytes выглядят как текст, состоящий из двоичных групп (например: "01111011 00100010 ..."), то нужно каждую группу из 8 бит преобразовать в байт и собрать исходную ASCII-строку.
- В JSON изменить поле is_admin на "True" (в примере значение представлено строкой).
- Преобразовать JSON обратно в строку и закодировать в base64.
- Подменить cookie в браузере на новое значение и обновить страницу.

Безопасность cookie — подробности

- Флаги cookie:
  - Secure: отправляет cookie только по HTTPS. Без него cookie может перехватиться в сети при HTTP.
  - HttpOnly: делает cookie недоступной для JavaScript (document.cookie), что защищает от некоторых XSS-атак.
  - SameSite: ограничивает отправку cookie кросс-сайтовых запросов (Lax/Strict/None), снижает риск CSRF.
  - Expires/Max-Age, Domain, Path — определяют область действия и срок жизни cookie.

- Целостность и подпись:
  - Если приложение хранит в cookie важные данные, оно должно их подписывать (например, HMAC-SHA256 с секретным ключом) или шифровать. Подпись позволяет серверу проверять, что данные не были изменены клиентом.
  - Альтернативный безопасный подход — хранить только идентификатор сессии в cookie, а все данные держать на сервере (в базе, кэше). Тогда изменение cookie на клиенте ничего не даст.

- Примеры атак и защита:
  - Изменение ролей/баланса в неподписанном cookie → получение прав администратора или денег.
  - Подделка cookie сессии без HttpOnly/secure → кража через XSS/перехват.

Примерный алгоритм эксплуатации (этапы)

1) Войти под обычным пользователем (user).
2) Открыть DevTools → Application → Cookies и скопировать значение cookie secret.
3) Декодировать base64 (можно с помощью python или онлайн-инструмента).
4) Если после декодирования получился JSON — изменить is_admin на "True".
   Если получилась строка с двоичными группами — преобразовать группы в байты, получить JSON, изменить поле и при необходимости вернуть в тот же формат (либо прямой JSON в base64 — зависит от того, как проверяет сервер).
5) Преобразовать обратно в base64, заменить значение cookie и обновить страницу.

Примечание: в реальной проверке нужно убедиться, что сервер не проверяет подпись/целостность. Если подпись есть — простой подход не сработает. Тогда ищут утечки ключа, уязвимости в алгоритме подписи или обходы в логике аутентификации.
